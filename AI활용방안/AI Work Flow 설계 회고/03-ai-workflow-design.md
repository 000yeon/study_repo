# Part 3 — AI 워크플로우 설계: 13개 컨텍스트 문서와 5가지 시나리오

> AI가 프로젝트를 모르는 것이 문제라면, 프로젝트를 알게 만들면 된다.

---

## 설계 철학: "컨텍스트가 품질을 결정한다"

AI 코드 생성 도구를 사용하면서 얻은 가장 중요한 인사이트는 하나다.

**AI의 출력 품질은 입력 컨텍스트의 품질과 정비례한다.**

"공지사항 CRUD 만들어줘"와 "이 프로젝트의 디렉토리 구조, 라우팅 패턴, 폼 처리 방식, 쿼리 패턴을 읽은 후 공지사항 CRUD를 만들어줘"의 결과물은 완전히 다르다.

전자는 AI가 최선의 추측으로 만든 코드다. 후자는 우리 프로젝트에 맞는 코드다.

이 워크플로우는 **"후자의 컨텍스트를 패키지로 만들어 팀 전체가 재사용하는 시스템"** 이다.

---

## Template 시스템: 13개 문서의 역할 분리

컨텍스트 문서를 하나의 거대한 파일에 넣는 것은 효율적이지 않다. AI의 컨텍스트 윈도우를 낭비하고, 작업에 필요 없는 정보까지 불러오게 된다.

13개 문서로 분리한 이유는 **상황에 따라 필요한 문서만 선택해서 줄 수 있기 때문**이다.

| 문서 | 역할 | 항상 필요 |
|------|------|----------|
| 01-overview | 프로젝트 목적, 기술 스택, 팀 규칙 | ✓ |
| 02-project-setup | package.json, tsconfig, 환경변수 | 신규 프로젝트 시 |
| 03-directory-structure | 폴더 구조와 파일 배치 규칙 | ✓ |
| 04-routing-and-layout | 라우트 구조, 레이아웃 컴포넌트 | 도메인 추가 시 |
| 05-design-system | 색상 토큰, 타이포그래피, 간격 | UI 구현 시 |
| 06-component-inventory | 공통 컴포넌트 목록과 Props | UI 구현 시 |
| 07-query-patterns | API 호출, React Query 패턴 | 항상 |
| 08-form-patterns | 폼 구현, 유효성 검사 패턴 | 폼 작업 시 |
| 09-domain-guide | 새 도메인 추가 21단계 체크리스트 | 도메인 추가 시 |
| 10-auth-and-middleware | 인증 흐름, 토큰 관리, 미들웨어 | 인증 관련 시 |
| 11-state-management | 전역 상태, 필터 상태 패턴 | 상태 관련 시 |
| 12-coding-conventions | 네이밍, import 순서, 주석 규칙 | ✓ |
| 13-figma-workflow | Figma MCP 연동 절차 | Figma 작업 시 |

### AI_CONTEXT/ vs docs/template/ 분리

두 폴더를 분리한 데는 이유가 있다.

- **`docs/template/`** — 패턴과 규칙을 정의하는 **아키텍처 문서**. 프로젝트 초기에 한 번 작성하고, 구조가 바뀔 때만 수정한다.
- **`AI_CONTEXT/`** — 매 작업에서 AI에게 주는 **운영 문서**. 도메인이 추가되거나 API가 바뀔 때 함께 갱신된다.

이 분리 덕분에 "이번 작업에 어떤 문서를 줘야 하는가"를 결정하는 비용이 줄었다. AI_CONTEXT의 파일들은 항상 최신 상태를 유지하고, template은 한 번 만들면 재사용된다.

## 5가지 시나리오: 상황에 따른 문서 조합

모든 작업에 13개 문서를 전부 주는 것은 비효율적이다. 작업 유형에 따라 최적의 문서 조합이 다르다.

### 시나리오 C — 신규 프로젝트 구성

새 프로젝트를 처음 만들 때. 구조를 잡는 것이 목적이므로 가장 많은 문서가 필요하다.
```
01 + 02 + 03 + 04 + 07 + 08 + 10 + 11 + 12 + 13
```

### 시나리오 A — 도메인 CRUD 추가 (가장 자주 쓰는 패턴)

공지사항, 이벤트, 상품처럼 새 도메인을 추가할 때. 구조와 패턴 문서에 집중한다.
```
01 + 03 + 06 + 07 + 08 + 09 + 12 (+ Figma URL이 있으면 13)
```

### 시나리오 B — Mock → 실제 API 연동

Mock으로 만든 화면을 실제 API와 연결할 때. View는 건드리지 않고 Query layer만 바꾼다.
```
01 + 07
```

### 시나리오 D — 버그 수정 / 기능 수정

특정 화면의 문제를 고칠 때. 관련 패턴 문서만 선택한다.
```
01 + 관련 패턴 문서 (07 or 08 or 11)
```

### 시나리오 E — 리팩토링

도메인 구조 통일, Mock 위치 정리, 공통 UI 중복 제거 등. 구조와 컨벤션 문서를 기준으로 한다.
```
03 + 06 + 07 + 09 + 12
```

## Figma MCP 연동: 디자인-코드 갭 해소

기존 워크플로우에서 가장 큰 품질 손실이 일어나는 지점이 **Figma → 코드** 전환이었다.

개발자가 Figma를 보고 직접 색상값과 간격을 입력하는 방식은:
1. 오차가 발생한다 (HEX 코드를 눈으로 복사)
2. 시간이 걸린다 (토큰을 하나하나 확인)
3. 업데이트가 안 된다 (디자인이 바뀌면 코드도 수동으로 바꿔야)

Figma MCP(Model Context Protocol)는 Claude가 Figma 파일을 직접 읽을 수 있게 한다.

### 워크플로우

```
1. Figma 파일 URL 제공
   ↓
2. Claude가 design context 추출
   → 색상 토큰, 타이포그래피, 컴포넌트 구조 자동 분석
   ↓
3. 추출한 토큰으로 05-design-system.md 재생성
   ↓
4. 컴포넌트 목록으로 06-component-inventory.md 재생성
   ↓
5. 이후 모든 UI 구현 시 이 두 문서를 기준으로 코드 작성
```

이 방식을 도입한 후 Figma 컬러 토큰 오차율이 **18% → 3% 미만**으로 줄었다.

## [PROJECT_ROOT] 추상화: 이식성 확보

초기 버전의 프롬프트에는 프로젝트 절대 경로가 하드코딩되어 있었다.

```
/Users/developer/Desktop/projects/moda/moda-frontend-admin/AI_CONTEXT/...
```

이 방식은 두 가지 문제가 있었다:
1. 다른 팀원의 로컬 환경에서 경로가 달라진다
2. 다른 프로젝트에 이 템플릿을 재사용할 수 없다

`[PROJECT_ROOT]` 플레이스홀더로 추상화한 후, 팀원들이 처음 사용할 때 자신의 경로로 교체하면 된다. 프롬프트 템플릿의 **이식성**이 생겼다.

## Smoke Check 방법론

AI가 코드를 만든 후 검증 단계가 약했다. `tsc --noEmit`으로 타입 오류만 확인하고 브라우저에서 실제로 동작하는지 확인하지 않는 일이 있었다.

**Smoke Check**는 브라우저에서 핵심 흐름을 직접 눌러보는 최소한의 검증이다.

```
✓ 목록 URL 직접 접근 → 데이터가 렌더링되는가
✓ 빈 상태(데이터 없음) → NoData 컴포넌트가 뜨는가
✓ 등록 버튼 클릭 → 폼이 열리고 저장 후 알림이 뜨는가
✓ 수정 화면 진입 → 기존 값이 채워져 있는가
✓ 다른 탭으로 이동 → 미저장 경고(leave guard)가 뜨는가
✓ 삭제 버튼 클릭 → 확인 다이얼로그 후 삭제되는가
```

이 체크리스트를 워크플로우의 마지막 단계로 의무화한 후, "빌드는 됐는데 화면이 안 뜬다" 류의 이슈가 사라졌다.

## 설계에서 얻은 가장 중요한 교훈

> **"AI에게 바로 구현해줘를 던지지 않는다"**

이 한 줄이 이 워크플로우 전체를 관통하는 원칙이다.

항상 컨텍스트 문서를 먼저 주고, AI가 계획을 먼저 보여주게 하고, 계획을 확인한 후 구현을 시작한다. 이 순서를 지키는 것만으로도 재작업률이 크게 달라진다.

결과는 [Part 4](./04-results-and-metrics.md)에서 확인할 수 있다.
