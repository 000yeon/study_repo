# Part 4 — 팀 도입 결과와 수치: 운영 6주 기준

> 숫자는 거짓말을 하지 않는다. 하지만 맥락 없는 숫자는 의미가 없다.

---

## 운영 환경

- **팀 구성**: 팀장 1명 + 개발자 4명 + PM 1명 (총 6명)
- **프로젝트**: Admin 프로젝트 3개 동시 운영
- **AI 도구**: Claude Code (주), GitHub Copilot (보조)
- **측정 기간**: 워크플로우 도입 후 6주

비교 기준은 도입 전 6주 동안 같은 팀이 같은 규모의 작업을 수행한 내역이다.

## 지표별 상세

### 1. 신규 도메인 CRUD: 3.5일 → 1.5일

도입 전, 새 도메인을 추가할 때 걸린 평균 시간은 **3.5일**이었다. 여기에는 다음 과정들이 포함된다:

- 기존 도메인의 파일 구조를 찾아보고 참조하는 시간
- AI에게 컨텍스트를 설명하는 시간
- AI가 만든 코드를 프로젝트 구조에 맞게 수정하는 시간
- 타입 오류를 잡는 시간
- 화면 동작을 확인하는 시간

도입 후 같은 과정이 **1.5일**로 줄었다. 감소한 시간의 대부분은 "컨텍스트 설명"과 "코드 수정" 에서 나왔다. 컨텍스트 문서를 한 번 넘기면 AI가 처음부터 프로젝트 구조에 맞는 코드를 만들었다.

6명 팀이 6주 동안 신규 도메인을 총 11개 추가했다. 도메인당 2일 절약 × 11개 = **팀 전체 22인일(person-day) 절약**.

---

### 2. AI 재작업률: 38% → 9%

"재작업"은 AI가 생성한 코드를 **30% 이상 수정해야 하는 경우**로 정의했다.

도입 전, AI가 만든 코드의 38%는 의미 있는 수정이 필요했다:
- 잘못된 디렉토리에 파일 생성
- 프로젝트에서 사용하지 않는 라이브러리 import
- 이미 있는 공통 컴포넌트 대신 인라인으로 새로 구현
- 쿼리 패턴이 프로젝트 기준과 다름

도입 후 재작업률은 **9%**로 줄었다. 남은 9%는 주로 API 스펙이 Figma와 맞지 않는 경우 — AI 문제가 아니라 기획/디자인 문제였다.

---

### 3. 환경 세팅 온보딩: 4시간 → 25분

신규 팀원이 프로젝트에 합류했을 때 개발 환경을 세팅하는 데 걸린 시간이다.

도입 전:
- "어떤 Node.js 버전 써야 해요?" → 구두로 전달
- "yarn이에요 npm이에요?" → 구두로 전달
- ESLint 설정이 왜 안 되는지 한참 디버깅
- Husky가 없어서 커밋 형식을 처음에 모름

도입 후:
- `Environment/README.md`의 빠른 시작 5단계를 따르면 끝
- 5단계 실행 시간 실측: **평균 23분**

팀에서 6주 동안 신규 합류 인원이 2명 있었다. 1인당 3.5시간 절약 × 2명 = **7시간 절약**. 수치보다 더 중요한 것은, 신규 팀원이 합류 첫날부터 막히지 않고 개발을 시작할 수 있었다는 점이다.

---

### 4. PR 포맷/컨벤션 코멘트 비율: 28% → 4%

PR 리뷰 코멘트를 유형별로 분류했을 때, 도입 전에는 전체 코멘트의 **28%**가 "포맷"이나 "컨벤션" 관련이었다.

```
# 도입 전 흔한 리뷰 코멘트
- "import 순서 맞춰주세요"
- "세미콜론 빠졌어요"
- "Tailwind 클래스 순서 통일해주세요"
- "커밋 메시지 형식이 달라요"
- "console.log 지워주세요"
```

Husky pre-commit + ESLint + Prettier를 도입한 후, 이런 코멘트들이 커밋 단계에서 이미 처리된다. 리뷰어가 포맷 대신 **로직과 설계**에 집중할 수 있게 됐다.

---

### 5. CI 린트/포맷 실패: 주 2.3건 → 0건

도입 전, CI 파이프라인에서 린트나 포맷 오류로 빌드가 실패하는 일이 주 평균 **2.3건** 있었다. 이는 PR을 올렸다가 CI 결과를 기다리고, 실패를 확인하고, 수정하고, 다시 push하는 사이클을 의미한다.

Husky + lint-staged로 커밋 전에 이미 처리되므로, 이 사이클이 완전히 사라졌다.

---

### 6. Figma 컬러 토큰 오차: ~18% → 3% 미만

디자이너가 지정한 색상 토큰과 코드에서 사용된 색상값의 일치율 측정 결과다.

도입 전, 개발자가 Figma를 보고 수동으로 HEX 코드를 입력하는 방식에서 약 **18%의 토큰에 오차**가 있었다 (HEX 코드 오타, 비슷한 색상 혼동, 업데이트 누락).

Figma MCP로 Claude가 직접 Figma 파일을 읽고 `05-design-system.md`를 생성한 후, 오차율이 **3% 미만**으로 낮아졌다. 남은 3%는 Figma의 오버레이 색상처럼 MCP가 추출하지 못하는 특수 케이스였다.

## 예상하지 못한 부가 효과

### 문서가 살아있는 아키텍처 문서가 됐다

처음에는 AI를 위한 컨텍스트 문서를 만들었다. 그런데 이 문서들이 팀 전체의 **아키텍처 참조 문서**로 자연스럽게 기능하게 됐다. 새 팀원이 "이 프로젝트에서 쿼리는 어떻게 작성해요?"라는 질문을 할 때, 이제 `07-query-patterns.md`를 링크로 공유하면 됐다.

### AI 프롬프트 작성 능력이 팀 전체에 퍼졌다

워크플로우가 없을 때는 AI 활용 수준이 팀원마다 달랐다. 잘 쓰는 사람은 잘 쓰고, 모르는 사람은 아예 안 쓰거나 기본적인 것만 물어봤다. 표준 프롬프트 패턴(`HOW-TO-USE-AI.html` Section 04)을 공유한 후, 팀 전체의 AI 활용 빈도와 품질이 평준화됐다.

### PM이 기능 명세를 더 구체적으로 쓰게 됐다

`FEATURES/` 폴더에 도메인별 작업 지시 문서를 작성하는 프로세스가 생기면서, PM이 개발자에게 전달하는 기능 명세의 디테일이 올라갔다. AI에게 전달할 명세를 쓰다 보면, 기획의 구멍이 먼저 보이기 때문이다.
