# Part 5 — 회고: 잘한 것, 아쉬운 것, 다음 스텝

> 좋은 시스템은 처음부터 완벽하지 않다. 운영하면서 배운 것들이 시스템을 완성한다.

---

## 잘한 것

### 1. 문서를 코드처럼 버전 관리했다

AI 컨텍스트 문서를 별도 레포에서 Git으로 관리한 것은 좋은 선택이었다.

- 문서가 언제 왜 바뀌었는지 `git log`로 추적할 수 있다
- PR로 문서 변경을 리뷰할 수 있다
- 잘못된 변경을 `git revert`로 되돌릴 수 있다
- 다른 프로젝트에 `git clone`으로 가져갈 수 있다

"문서는 Word나 Notion에"라는 관행을 깨고 Markdown + Git을 선택한 것이 이 시스템의 내구성을 만들었다.

### 2. 개발 환경 표준화가 팀 마찰을 없앴다

pnpm + ESLint + Prettier + Husky 세트를 `Environment/` 문서로 표준화한 후, "내 환경에서는 되는데 왜 CI가 실패하죠?"나 "ESLint 오류가 너무 많이 나요"같은 문제가 완전히 사라졌다.

특히 Husky의 pre-commit 훅이 생각보다 강력했다. 팀원들이 별도로 ESLint를 신경 쓰지 않아도 커밋 전에 자동으로 수정되니, 개발 흐름을 방해하지 않으면서 코드 품질이 유지됐다.

### 3. 시나리오 분류가 AI 사용 진입 장벽을 낮췄다

5가지 시나리오(A~E)로 "어떤 상황에 어떤 문서를 줘야 하는가"를 명확히 했다. 이 분류가 없었다면 팀원마다 다른 방식으로 AI를 쓰게 됐을 것이다.

시나리오 A(도메인 추가)가 전체 사용의 약 60%를 차지했다. 이 시나리오의 프롬프트와 문서 조합이 가장 다듬어진 것이 전체 품질의 핵심이었다.

### 4. Figma MCP 연동이 디자인-코드 갭을 실질적으로 줄였다

초기에는 "Figma MCP가 얼마나 실용적으로 쓰일지" 반신반의했다. 실제 도입 후 색상 토큰 오차 문제가 사실상 해결됐고, 디자이너와 개발자 사이의 QA 라운드가 크게 줄었다.

## 아쉬운 것

### 1. 초기 문서 작성에 투자한 시간이 많았다

워크플로우 문서 전체를 처음 만드는 데 약 **40시간**이 들었다. 13개 Template 문서, Sample/ 파일들, Environment/ 문서들, HOW-TO-USE-AI.html 등. 이 초기 비용이 작은 팀이나 빠른 납기 프로젝트에서는 걸림돌이 될 수 있다.

다음 프로젝트부터는 이 레포를 그대로 복사해서 쓸 수 있으므로 초기 비용은 이미 상각됐지만, 처음 시작하는 팀에게는 진입 장벽이다.

**개선 방향**: 프로젝트 유형별로 기본값을 제공하는 CLI 초기화 도구를 만들면 초기 비용을 2~4시간으로 줄일 수 있을 것이다.

### 2. Template 05/06 문서를 매 프로젝트마다 재생성해야 한다

디자인 시스템(05)과 컴포넌트 목록(06)은 프로젝트마다 다르다. Figma MCP로 자동화했지만, 프로젝트마다 Figma 파일 구조가 다르기 때문에 추출 결과를 항상 검토하고 수동 보정이 필요했다.

특히 Figma에서 컴포넌트를 잘 정리하지 않은 프로젝트는 MCP 추출 결과가 혼란스러웠다. **Figma 파일 구조 가이드**를 디자이너에게 미리 제공하는 것이 해결책이다.

### 3. AI가 항상 컨텍스트를 완벽하게 소화하지 못한다

13개 문서를 모두 줬을 때 Claude의 컨텍스트 윈도우에 따라 뒤쪽 문서 내용이 덜 반영되는 경우가 있었다. 특히 12-coding-conventions.md의 세부 규칙이 무시되는 일이 간헐적으로 발생했다.

이것은 AI 모델의 한계이기도 하고, 문서 설계의 한계이기도 하다. 중요한 규칙일수록 문서 앞쪽에 배치하거나, 프롬프트에 강조를 추가해야 한다. 현재는 `01-overview.md`에 "가장 중요한 규칙 3가지"를 명시하는 방식으로 보완했다.

### 4. PM의 FEATURES 문서 작성 부담

FEATURES 폴더에 도메인별 작업 지시 문서를 쓰는 것이 AI 품질을 높이는 핵심이었지만, PM에게 초기에 부담이 됐다. 어떤 수준의 디테일이 필요한지 처음에 감을 잡기 어려워했다.

`FEATURES/_TEMPLATE.md` 템플릿을 제공하고, 예시 파일을 같이 보여준 후 빠르게 적응했다. 초기 2주가 지나자 PM이 먼저 FEATURES 문서를 작성하고 개발자에게 전달하는 흐름이 자연스럽게 됐다.

## 다음 스텝

### 단기 (1~3개월)

**1. CLI 초기화 도구 개발**

```bash
npx create-admin-workflow my-project
```

이 한 줄로 Template/, Sample/, Environment/ 파일이 자동으로 세팅되고, `[PROJECT_ROOT]`가 자동으로 교체되도록 만들고 싶다.

**2. CI 파이프라인 문서 표준화**

GitHub Actions 워크플로우가 Environment/05-husky.md에 예시로만 있다. 실제 사용 가능한 `.github/workflows/ci.yml` 파일을 Template에 포함시켜 바로 복사해 쓸 수 있게 한다.

### 중기 (3~6개월)

**3. 문서 자동 갱신 훅**

API 스펙이 바뀌면 `07-query-patterns.md`가 outdated 된다. PR 머지 시 특정 파일이 변경됐을 때 관련 AI_CONTEXT 문서를 갱신하라는 알림이 가도록 자동화하고 싶다.

**4. 다른 도메인으로 확장**

현재 이 워크플로우는 Admin Frontend에 특화돼 있다. 같은 방식을 백엔드(NestJS, Spring), 모바일(React Native, Flutter)에도 적용할 수 있는지 탐색할 계획이다.

## 마치며

이 워크플로우를 만들면서 가장 크게 배운 것은 **시스템 설계의 가치**다.

AI 도구 자체는 계속 발전하고 있다. Claude 4, GPT-5, 그 다음 모델이 나올 것이다. 하지만 "AI에게 어떻게 우리 프로젝트를 이해시킬 것인가"라는 문제는 어떤 모델이 나와도 동일하게 존재한다.

컨텍스트 시스템을 한 번 잘 만들어두면, AI 모델이 업그레이드될 때마다 더 좋은 결과를 자동으로 얻을 수 있다. 이것이 이 작업에 40시간을 투자한 이유이고, 그 선택이 옳았다고 생각한다.
