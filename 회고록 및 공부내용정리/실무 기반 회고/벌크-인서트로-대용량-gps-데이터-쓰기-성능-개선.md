# 트러블슈팅 1: 벌크 인서트로 대용량 GPS 데이터 쓰기 성능 개선

## 배경

Tracky는 차량을 운영하는 기업을 위한 통합 관제 플랫폼이다. 차량의 준실시간 위치, 운행 현황, 주행 경로, 차량별 운행 기록을 제공하기 때문에 GPS 이력 데이터가 지속적으로 쌓인다.

처음에는 기능 구현에 집중해 위치 데이터가 들어올 때마다 개별 Insert로 저장했다. 샘플 차량 몇 대로 테스트할 때는 문제가 보이지 않았다. 하지만 목표 규모를 기준으로 다시 계산하니 구조적인 위험이 드러났다.

- 최대 차량 수: 15,000대
- 데이터 수집 주기: 1초
- 서버 처리 단위: 60초
- 1분 누적 데이터: 15,000대 * 60초 = 900,000건

개별 Insert 방식이라면 1분마다 최대 900,000번의 Insert 쿼리가 발생한다. 이때 병목은 반복문 자체가 아니라 매 쿼리마다 발생하는 네트워크 왕복, SQL 파싱, 트랜잭션 처리, 인덱스 갱신 비용이었다.

## 문제 정의

처음에는 서버 로직이 느린 것인지, DB 저장이 느린 것인지 확신할 수 없었다. 그래서 데이터 생성, DTO 변환, 저장 호출, DB 반영 구간을 나누어 시간을 측정했다. 그 결과 비즈니스 로직보다 DB 쓰기 구간이 더 큰 비중을 차지했다.

이때 세운 가설은 다음과 같았다.

> 데이터 1건마다 Insert 쿼리를 1번씩 실행하는 구조에서는 데이터가 많아질수록 DB 왕복 비용이 선형적으로 증가한다. 따라서 쿼리 호출 횟수를 줄이면 전체 처리 속도를 개선할 수 있다.

즉, 목표는 단순히 "저장 코드를 빠르게 만들기"가 아니라 "900,000건 저장 과정에서 DB에 전달되는 요청 수를 줄이기"였다.

## 조사 과정

Bulk Insert를 검토하면서 단순히 여러 행을 한 번에 넣으면 빠르다는 수준에서 끝내지 않았다. 실제로 중요한 것은 어느 정도 크기로 묶어야 하는지였다.

조사하며 확인한 내용은 다음과 같다.

- Insert 쿼리 횟수가 줄면 애플리케이션과 DB 사이의 네트워크 왕복 비용이 줄어든다.
- 여러 행을 하나의 트랜잭션으로 처리하면 커밋 비용을 줄일 수 있다.
- 너무 큰 배치 크기는 SQL 길이 증가, 메모리 사용량 증가, 트랜잭션 유지 시간 증가를 만들 수 있다.
- JPA의 `saveAll`을 사용한다고 항상 실제 다건 Insert가 발생하는 것은 아니다.
- ORM, JDBC batch 설정, DB dialect에 따라 실제 쿼리 형태를 확인해야 한다.

이 조사를 통해 Bulk Insert 자체보다 "현재 시스템에서 가장 적절한 배치 크기를 찾는 것"이 핵심이라고 판단했다.

## 실험 설계

실험은 실제 운영 목표와 최대한 비슷하게 구성했다.

- 총 데이터: 900,000건
- 서버 사양: 2 vCPU, 10 thread, 4GB RAM
- DB 사양: AWS RDS t4g.small, 2 vCPU, 2GB RAM
- 비교 기준: 평균 처리 속도, Insert Query 횟수

비교한 방식은 다음과 같다.

| 방식 | 배치 크기 | Insert Query 횟수 | 실험 의도 |
| --- | ---: | ---: | --- |
| 개별 Insert | 1개 | 900,000번 | 초기 구현 방식 검증 |
| Bulk Insert | 60개 | 15,000번 | 차량 1대의 1분 데이터 단위 |
| Bulk Insert | 600개 | 1,500번 | 여러 차량 데이터를 적당히 묶는 방식 |
| Bulk Insert | 15,000개 | 60번 | 1초당 전체 차량 데이터를 한 번에 묶는 방식 |

처음 예상은 단순했다. 많이 묶을수록 쿼리 수가 줄어드니 더 빠를 것이라고 생각했다. 하지만 너무 큰 배치가 오히려 SQL 처리 부담과 메모리 부담을 만들 수 있다는 점 때문에, 배치 크기별 비교가 필요했다.

## 실험 결과

| 방식 | Insert Query 횟수 | 평균 처리 속도 |
| --- | ---: | ---: |
| 1개씩 Insert | 900,000번 | 128ms |
| 60개씩 Bulk Insert | 15,000번 | 106ms |
| 600개씩 Bulk Insert | 1,500번 | 76ms |
| 15,000개씩 Bulk Insert | 60번 | 85ms |

가장 흥미로운 결과는 15,000개씩 묶는 방식이 가장 빠르지 않았다는 점이다. 쿼리 횟수만 보면 60번으로 가장 적지만, 실제 평균 처리 속도는 600개씩 묶는 방식보다 느렸다.

이 결과로 다음을 확인했다.

- 쿼리 횟수를 줄이는 것은 분명히 효과가 있다.
- 하지만 배치 크기를 무작정 키우면 한 번의 쿼리가 너무 무거워질 수 있다.
- 최적점은 쿼리 횟수 감소 효과와 단일 쿼리 부담이 균형을 이루는 지점에 있다.

## 최종 적용

Tracky에서는 600개 단위 Bulk Insert를 최종 선택했다. 실험상 가장 빠르기도 했고, 장애 대응 관점에서도 적절했다.

최종 흐름은 다음과 같이 정리했다.

```text
GPS 데이터 수신
  -> 메시지 또는 내부 버퍼에 적재
  -> 600개 단위 chunk 생성
  -> Bulk Insert 실행
  -> 실패 시 chunk 단위 재처리
```

600개 단위는 단순히 빠른 숫자가 아니라 운영상 균형이 좋은 크기였다. 너무 작은 배치는 쿼리 횟수가 많고, 너무 큰 배치는 실패 시 재처리 부담이 크다. 600개 단위는 처리 속도, 메모리 부담, 실패 복구 범위가 모두 관리 가능한 수준이었다.

## 개선 효과

```text
개선 전: 128ms
개선 후: 76ms
개선율: 약 41%
```

Insert Query 횟수도 900,000번에서 1,500번으로 줄었다. 이 개선은 애플리케이션 처리 속도뿐 아니라 DB 부하, 네트워크 왕복, 트랜잭션 비용까지 함께 줄인 결과였다.

## 회고

이번 경험에서 가장 크게 배운 점은 성능 개선은 유명한 기술을 적용하는 것이 아니라 병목을 측정하고 그 병목에 맞는 해결책을 검증하는 과정이라는 것이다.

처음부터 Bulk Insert를 맹목적으로 선택한 것이 아니라, 개별 Insert 구조의 한계를 수치로 확인했고, 배치 크기별로 비교했으며, 600개 단위가 가장 적절하다는 결론을 실험으로 얻었다.

특히 "쿼리 횟수가 가장 적은 방식이 항상 가장 빠르지는 않다"는 점을 직접 확인한 것이 의미 있었다. 이후 성능 문제를 볼 때도 직관보다 측정 결과와 실행 환경을 우선하게 되었다.
