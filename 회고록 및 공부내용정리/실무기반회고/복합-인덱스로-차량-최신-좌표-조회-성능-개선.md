# 트러블슈팅 3: 복합 인덱스로 차량 최신 좌표 조회 성능 개선

## 배경

Tracky에서 가장 중요한 조회 기능 중 하나는 차량별 최신 좌표를 가져오는 API다. 대시보드나 지도 화면에서는 사용자가 현재 차량들이 어디에 있는지 빠르게 확인해야 한다.

하지만 초기 테스트에서 차량별 최신 좌표 조회 API가 4초 이상 걸리는 문제가 발생했다. 준실시간 관제 서비스에서 지도 화면을 열 때마다 몇 초씩 기다려야 한다면 서비스의 핵심 가치가 약해진다.

## 문제 정의

먼저 API 내부 로직을 구간별로 나누어 측정했다.

| 구간 | 비중 |
| --- | ---: |
| 요청 파라미터 처리 | 낮음 |
| 권한 및 업체 정보 조회 | 낮음 |
| 차량별 최신 좌표 조회 쿼리 | 약 94% |
| 응답 DTO 변환 | 낮음 |

측정 결과 전체 응답 시간의 대부분이 조회 쿼리에서 발생했다. 따라서 애플리케이션 코드를 미세하게 최적화하는 것보다 쿼리 실행 계획과 인덱스를 분석하는 것이 우선이라고 판단했다.

## 기존 쿼리의 문제

초기 쿼리는 차량별 최신 운행 데이터와 GPS 이력을 함께 조회하는 구조였다.

```sql
SELECT gh.lat, gh.lon
FROM gpshistory gh
JOIN drive d ON gh.drive_id = d.id
WHERE d.id IN (
    SELECT MAX(d2.id)
    FROM drive d2
    JOIN car c2 ON d2.mdn = c2.mdn
    WHERE c2.biz_id = 1
    GROUP BY d2.mdn
)
GROUP BY d.id, d.mdn
ORDER BY drive_seq DESC;
```

의도는 명확했다. 업체에 속한 차량들을 찾고, 차량별 최신 drive를 찾은 뒤, 해당 drive의 최신 GPS 좌표를 가져오는 것이다.

하지만 실행 계획을 확인하자 병목이 명확하게 보였다.

| table | type | key | rows | Extra |
| --- | --- | --- | ---: | --- |
| gps_history | ALL | null | 1,024,056 | Using where; Using Temporary; Using filesort |

가장 큰 문제는 `gps_history`에서 풀 테이블 스캔이 발생한다는 점이었다. GPS 이력 테이블은 데이터가 가장 빠르게 증가하는 테이블이다. 이 테이블에서 100만 건 이상을 스캔하고, 임시 테이블과 filesort까지 사용하면 응답 시간이 길어질 수밖에 없었다.

## 병목 원인 분석

실행 계획을 기준으로 병목을 다음처럼 정리했다.

1. 가장 큰 테이블인 `gps_history`에서 인덱스를 제대로 활용하지 못했다.
2. 차량별 최신 값을 찾기 위해 `GROUP BY`, `MAX`, `ORDER BY`가 함께 사용되었다.
3. 조인과 그룹화가 큰 테이블 위에서 한꺼번에 수행되었다.
4. `Using Temporary`, `Using filesort`가 발생해 I/O 비용이 커졌다.
5. 최신 좌표 1건만 필요하지만 DB는 많은 후보 데이터를 훑고 정렬해야 했다.

핵심 문제는 "많이 읽은 뒤 원하는 결과로 줄이는 구조"였다. 더 좋은 방식은 DB가 인덱스를 타고 필요한 범위만 읽도록 쿼리와 인덱스를 맞추는 것이었다.

## 대안 검토

조회 성능 개선을 위해 여러 방법을 비교했다.

| 대안 | 장점 | 한계 |
| --- | --- | --- |
| 애플리케이션 캐싱 | 응답 속도 개선 가능 | 최신성 보장과 무효화 전략 필요 |
| Redis에 최신 좌표 저장 | 지도 조회에는 매우 빠름 | DB 이력과 Redis 동기화 실패 처리 필요 |
| 쿼리만 단순화 | 적용 부담 낮음 | 인덱스 없이 근본 개선 어려움 |
| 단일 컬럼 인덱스 추가 | 구현 쉬움 | `drive_id + drive_seq` 조건을 함께 최적화하기 어려움 |
| 복합 인덱스 추가 | 쿼리 패턴에 직접 대응 | 쓰기 시 인덱스 갱신 비용 증가 |

Redis로 최신 좌표를 따로 관리하는 방식도 매력적이었다. 하지만 당시 문제의 본질은 DB 조회 자체가 비효율적이라는 점이었다. DB 쿼리가 4초 이상 걸리는 상태에서 캐시를 덧씌우면 원인을 가리는 선택이 될 수 있다고 봤다.

따라서 먼저 DB 조회 구조를 개선하고, 캐시는 트래픽이 더 커졌을 때 추가로 검토하기로 했다.

## 복합 인덱스를 선택한 이유

조회 API가 원하는 데이터는 단순했다.

> 특정 drive에 속한 GPS 이력 중 drive_seq가 가장 큰 최신 좌표

즉, GPS 이력 테이블에서 자주 함께 사용되는 컬럼은 `drive_id`와 `drive_seq`였다.

- `drive_id`: 어떤 운행 기록에 속한 GPS 이력인지 찾는 기준
- `drive_seq`: 해당 운행 안에서 시간 순서를 나타내는 기준

따라서 다음 복합 인덱스가 조회 패턴에 가장 잘 맞는다고 판단했다.

```sql
CREATE INDEX idx_gpshistory_drive_id_drive_seq
ON gpshistory (drive_id, drive_seq);
```

이 인덱스가 적합했던 이유는 다음과 같다.

1. `WHERE` 또는 `JOIN`에서 `drive_id`로 범위를 좁힐 수 있다.
2. 같은 `drive_id` 안에서 `drive_seq` 순서로 정렬된 데이터를 활용할 수 있다.
3. `GROUP BY drive_id`와 `MAX(drive_seq)` 패턴에서 인덱스 활용 가능성이 커진다.
4. 최신 좌표를 찾기 위해 전체 테이블을 정렬하지 않아도 된다.
5. 조인 시 필요한 GPS 이력 1건에 더 가깝게 접근할 수 있다.

단일 인덱스가 아니라 복합 인덱스가 필요했던 이유도 여기에 있다. `drive_id`만 인덱스가 있으면 특정 운행의 GPS 이력을 찾는 데는 도움이 되지만, 그 안에서 최신 순번을 찾기 위해 추가 정렬이 필요할 수 있다. `drive_seq`만 인덱스가 있으면 특정 drive 범위를 빠르게 좁히기 어렵다.

## 쿼리 구조 개선

인덱스를 추가하는 것만으로는 충분하지 않았다. 쿼리도 DB가 인덱스를 활용하기 쉬운 구조로 바꿔야 했다.

개선 방향은 다음과 같았다.

1. 업체에 속한 차량의 최신 drive를 먼저 구한다.
2. 각 drive별 최신 `drive_seq`를 구한다.
3. `drive_id + drive_seq`로 GPS 이력의 정확한 1건을 조인한다.

개념적으로는 다음과 같은 CTE 구조로 정리했다.

```sql
WITH latest_drive AS (
    SELECT d.mdn, MAX(d.id) AS max_drive_id
    FROM drive d
    JOIN car c ON d.mdn = c.mdn
    WHERE c.biz_id = :bizId
    GROUP BY d.mdn
),
latest_seq AS (
    SELECT gh.drive_id, MAX(gh.drive_seq) AS max_seq
    FROM gpshistory gh
    JOIN latest_drive ld ON gh.drive_id = ld.max_drive_id
    GROUP BY gh.drive_id
)
SELECT gh.lat, gh.lon
FROM latest_drive ld
JOIN latest_seq ls ON ls.drive_id = ld.max_drive_id
JOIN gpshistory gh
  ON gh.drive_id = ls.drive_id
 AND gh.drive_seq = ls.max_seq;
```

이 구조의 목표는 큰 테이블을 한 번에 조인하고 정렬하는 것이 아니라 단계적으로 후보를 줄이는 것이다. 특히 마지막 조인에서 `(drive_id, drive_seq)` 복합 인덱스를 사용하면 필요한 좌표 한 건에 가깝게 접근할 수 있다.

## 실행 계획 비교

개선 후 실행 계획에서는 다음과 같은 변화가 있었다.

| table | type | key | rows | Extra |
| --- | --- | --- | ---: | --- |
| gps_history | range | `(drive_id, drive_seq)` | 224 | Using index for group-by |
| gps_history | eq_ref | PRIMARY | 1 | Using where |

기존에는 `gps_history`에서 1,024,056건 수준의 풀 스캔이 발생했다. 개선 후에는 인덱스를 활용해 조회 범위가 크게 줄었고, group-by에서도 인덱스를 활용할 수 있게 되었다.

특히 `Using index for group-by`가 확인된 점이 중요했다. 이는 DB가 그룹화 과정에서 인덱스를 활용하고 있다는 의미였고, 기존의 `Using Temporary`, `Using filesort`보다 훨씬 유리한 실행 방식이었다.

## 적용 후 고려한 점

복합 인덱스는 조회 성능을 크게 개선했지만 비용이 없는 선택은 아니다. GPS 이력 테이블은 쓰기가 매우 많은 테이블이기 때문에 인덱스를 추가하면 Insert 시 인덱스 갱신 비용이 늘어난다.

그래서 적용 후 다음을 함께 확인해야 한다고 정리했다.

- Bulk Insert 성능이 인덱스 추가 후에도 목표 범위 안에 있는지 확인
- 인덱스 크기가 과도하게 커지지 않는지 확인
- 실제 조회 패턴에서 사용되지 않는 인덱스를 만들지 않기
- 향후 파티셔닝이나 보관 정책 검토
- 최신 좌표 조회 빈도가 더 커지면 Redis 캐시 추가 검토

즉, 복합 인덱스는 모든 문제의 최종 답이라기보다 현재 병목에 가장 직접적으로 대응한 해결책이었다. 데이터가 더 커지고 조회 패턴이 바뀌면 다시 실행 계획을 확인해야 한다.

## 회고

이번 트러블슈팅에서 가장 값진 경험은 실행 계획을 직접 보고 병목을 해석한 것이다. 단순히 쿼리가 느리다고 느낀 것이 아니라 `ALL`, `Using Temporary`, `Using filesort`, `rows` 값을 보고 DB가 실제로 어떤 일을 하고 있는지 확인했다.

그리고 인덱스를 추가할 때도 "인덱스가 있으면 빨라진다"가 아니라 조회 패턴과 컬럼 순서를 기준으로 판단했다. Tracky의 경우 최신 좌표 조회는 `drive_id`로 범위를 좁히고 `drive_seq`로 최신 데이터를 찾는 구조였기 때문에 `(drive_id, drive_seq)` 복합 인덱스가 적합했다.

이 경험 이후 쿼리 최적화를 볼 때는 항상 다음 순서로 접근하게 되었다.

1. API 전체 응답 시간을 측정한다.
2. 애플리케이션 로직과 DB 쿼리 시간을 분리해 측정한다.
3. 느린 쿼리의 실행 계획을 확인한다.
4. 풀 스캔, 임시 테이블, filesort, 예상 rows를 본다.
5. 조회 패턴에 맞는 인덱스 후보를 세운다.
6. 인덱스 추가 전후 성능과 쓰기 비용을 함께 비교한다.
