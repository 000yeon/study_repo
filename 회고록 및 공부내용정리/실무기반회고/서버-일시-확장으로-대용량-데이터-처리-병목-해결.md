# 트러블슈팅 2: 서버 일시 확장으로 대용량 데이터 처리 병목 해결

## 배경

Tracky는 차량 GPS 데이터를 준실시간으로 수집하고 저장하는 차량 관제 플랫폼이다. 최대 15,000대 차량이 1초마다 위치 데이터를 보낸다고 가정하면, 1분 동안 처리해야 하는 데이터는 900,000건이다.

Bulk Insert를 적용해 DB 쓰기 속도는 개선했지만, 여전히 한 가지 질문이 남았다.

> 900,000건 데이터를 반드시 1분 안에 처리해야 한다면 애플리케이션 서버 한 대로 충분할까?

Tracky의 위치 데이터는 나중에 천천히 처리해도 되는 일반 로그가 아니다. 데이터 처리가 지연되면 대시보드의 차량 위치, 운행 상태, 최신 경로가 늦게 반영된다. 준실시간 관제 서비스에서는 처리 지연이 곧 사용자 경험 저하로 이어진다.

## 문제 정의

초기 인프라 조건은 다음과 같았다.

- 애플리케이션 서버: AWS Fargate
- 서버 사양: 2 vCPU, 4GB RAM, 10 thread
- DB: AWS RDS t4g.small
- DB 사양: 2 vCPU, 2GB RAM
- 목표: 900,000건 데이터를 60초 안에 처리

서버 한 대에서 처리량이 부족하다면 두 가지 선택지가 있었다.

1. 서버 한 대의 스펙을 높인다.
2. 같은 작업을 여러 서버가 나누어 처리하게 한다.

처음에는 Fargate task의 CPU와 메모리를 높이는 수직 확장을 생각했다. 하지만 GPS 데이터 처리 부하는 항상 일정하지 않았다. 차량 운행이 몰리는 시간대나 1분 단위 처리 시점에는 부하가 높고, 야간이나 운행량이 적은 시간에는 리소스가 남을 수 있다.

이 상황에서 최대 부하를 기준으로 서버를 항상 크게 유지하는 것은 비용 효율이 낮았다. 그래서 대량 데이터가 몰리는 구간에는 서버 수를 일시적으로 늘리고, 처리가 안정되면 다시 줄이는 수평 확장 전략을 검토했다.

## 조사 과정

서버 수를 늘리는 전략을 검토하면서 다음 질문을 먼저 정리했다.

1. 서버를 늘리면 실제 처리 시간이 줄어드는가?
2. DB가 병목이면 서버를 늘려도 효과가 제한되지 않는가?
3. 여러 서버가 동시에 처리할 때 중복 처리나 순서 문제가 생기지 않는가?
4. 비용 대비 성능 개선 폭이 충분한가?

조사하며 확인한 내용은 다음과 같다.

- CPU-bound 작업은 서버 수 증가가 비교적 직접적인 효과를 낼 수 있다.
- DB-bound 작업은 서버를 늘려도 DB 커넥션, I/O, 락 때문에 개선이 제한될 수 있다.
- MQ 기반 구조에서는 consumer 수를 늘려 처리량을 높일 수 있다.
- 메시지 중복 처리 가능성을 고려해 저장 작업은 idempotent하게 설계해야 한다.
- Auto Scaling 기준은 CPU 사용률뿐 아니라 queue backlog, 처리 지연 시간, DB connection 사용률도 함께 봐야 한다.

이 조사를 통해 서버 확장은 "무조건 늘리면 빨라진다"가 아니라, 작업을 안전하게 병렬화할 수 있고 DB가 감당할 수 있을 때 효과가 있다는 결론을 얻었다.

## 실험 설계

같은 조건에서 서버 수만 바꾸어 처리 시간을 비교했다.

- 처리 대상: 900,000건
- 목표 시간: 60초 이내
- 비교 대상: 서버 1대, 3대, 5대
- 서버별 사양: 2 vCPU, 4GB RAM, 10 thread
- DB 사양: RDS t4g.small, 2 vCPU, 2GB RAM

전체 처리 시간 외에도 다음 지표를 함께 확인했다.

- 서버별 처리량 편차
- DB connection 사용량
- 실패 또는 재시도 발생 여부
- 처리 중 CPU 사용률
- 데이터 누락 여부
- queue backlog 변화

## 실험 결과

| 서버 수 | 900,000건 처리 시간 | 목표 달성 여부 |
| ---: | ---: | --- |
| 1대 | 165초 | 실패 |
| 3대 | 80초 | 실패 |
| 5대 | 55초 | 성공 |

서버 1대에서는 165초가 걸려 목표인 60초를 크게 초과했다. 서버 3대로 늘렸을 때는 80초까지 줄었지만 여전히 목표에는 도달하지 못했다. 서버 5대에서는 55초로 줄어 1분 이내 처리 목표를 만족했다.

```text
1대: 165초
3대: 80초
5대: 55초

개선 전후 기준: 165초 -> 55초
개선율: 약 67%
```

## 결과 해석

서버 수를 1대에서 3대로 늘렸을 때 처리 시간이 크게 줄었다. 이는 Bulk Insert 준비, 데이터 분배, 저장 요청 생성 같은 애플리케이션 처리 구간이 병렬화의 영향을 받았다는 의미다.

하지만 3대에서 5대로 늘렸을 때 개선 폭은 상대적으로 줄었다. 이 지점이 중요했다. 서버를 늘릴수록 계속 선형적으로 빨라지는 것은 아니었다. 어느 순간부터는 DB 처리량, 커넥션 수, 네트워크, Insert 경합이 새로운 병목이 될 수 있다.

실험을 통해 얻은 결론은 다음과 같다.

- 서버 1대는 목표 처리량을 만족하지 못한다.
- 서버 3대는 개선 효과가 크지만 60초 목표에는 부족하다.
- 서버 5대는 목표를 만족하는 현실적인 최소 구성이다.
- 서버를 더 늘리기 전에는 DB 병목과 비용을 함께 검토해야 한다.

## 최종 적용 전략

최종적으로 서버를 항상 5대로 고정하는 방식보다, 부하 상황에 따라 일시적으로 늘렸다가 줄이는 전략이 더 적합하다고 판단했다.

| 상황 | 서버 전략 |
| --- | --- |
| 평상시 | 최소 task 수 유지 |
| 데이터 적재량 증가 | consumer task 수 증가 |
| queue backlog 증가 | scale-out |
| 처리 지연 시간 정상화 | scale-in |
| DB connection 임계치 접근 | 무리한 scale-out 제한 |

확장 기준은 CPU 사용률 하나로만 잡지 않는 것이 중요했다. GPS 데이터 처리에서는 CPU보다 queue backlog나 처리 지연 시간이 더 직접적인 신호가 될 수 있다.

예시 기준은 다음과 같다.

```text
queue backlog가 일정 시간 이상 증가
  -> 서버 task 수 증가

평균 처리 지연 시간이 60초 목표에 근접
  -> 서버 task 수 증가

DB connection 사용률이 임계치 이상
  -> 서버 증설보다 DB 병목 분석 우선

backlog가 해소되고 지연 시간이 안정화
  -> 서버 task 수 감소
```

## 회고

이번 트러블슈팅에서 가장 크게 배운 점은 성능 문제는 코드만의 문제가 아닐 수 있다는 것이다. Bulk Insert처럼 코드와 쿼리 구조를 개선하는 것도 중요하지만, 일정 규모 이상에서는 인프라 전략까지 함께 고려해야 했다.

90만 건을 1분 안에 처리한다는 목표는 기능 요구사항이면서 동시에 아키텍처 요구사항이었다. 단일 서버 최적화만으로는 부족했고, 작업을 병렬로 나눌 수 있는 구조가 필요했다.

또한 서버를 늘리는 선택은 쉬워 보이지만 실제로는 비용, DB 병목, 메시지 중복 처리, 장애 복구를 함께 고려해야 한다. 그래서 이번 실험의 결론은 단순히 "5대가 빠르다"가 아니라 "5대가 목표를 만족하는 최소 구성이고, 운영에서는 일시 확장 전략이 적합하다"였다.
